1

목적과 변경 요지

이번 변경에서 바뀌는 것은 단 하나, ‘보상 가격을 어떻게 취득하는가’입니다. 전체 4단계 중 1~3단계와 4.1은 기존 그대로 유지됩니다.

RFC란
Request for Credit — 워런티 기간 내 하자가 확인된 HDD 제품의 대금을 크레딧(Credit Note)으로 정산받는 절차. 본 문서에서는 업무 관행에 따라 ‘보상’으로 표기합니다.
요청
본사 SP 사업부
배경
HDD 시세와 환율이 판매 시점 이후 계속 오르면서 최신 견적가가 판매 당시 가격을 웃돌게 되었고, 이에 따라 보상 기준을 실제 판매가로 바로잡고자 함 (상세 배경은 5장 참조).
목적
HDD 아이템의 RFC(보상) 건에 대한 가격 적용 방식 변경
현재
RFC 해당 아이템의 최신 견적서 상 가격을 보상 단가로 사용
변경
해당 Serial No의 판매 인보이스 상 판매가격(Sales Price)을 보상 단가로 사용
최신 견적서 가격 → 판매 인보이스 판매가격
2

전체 업무 흐름

업무는 접수부터 본사 청구까지 네 단계로 이어지며, 이번에 가격 적용 방식이 바뀌는 곳은 마지막 ④단계 한 곳입니다.

① 하누 (서비스업체) — 접수 및 판정
업체로부터 교체/보상 접수 → 본사 RFC 사이트에서 Serial No 조회 → warranty in/out 판정
② RFC 접수
warranty in 인 경우 RFC 접수 및 RFC 번호 생성
③ TEKR(판매업체)로 송부
접수정보 파일인 RMA_Details 전달 — Serial No, Item, 구매업체, RFC 번호, 교체/보상 타입. 이 파일이 ④의 4.1에서 업로드하는 파일입니다.
④ TEKR 워크플로우(eSWiT) — 보상가격 적용 변경 영역
보상 타입(RFC)인 경우 사내 워크플로우 신청 → 가격 산정 → 본사 요청용 Debit Note 인보이스 송부
3

④단계 상세 — 현재 vs 변경

변경되는 내용은 모두 ④단계 안에 있습니다. 담당자가 수행하는 운영 작업과 IT가 구축하는 개발 사항을 구분해 표기했습니다.

기존 동일 신규 / 변경 담당자운영 작업 IT개발 사항 시스템수행 시스템 출처

현재 (AS-IS)

4.1 RMA_Details 파일 업로드 파일서버③에서 받은 접수정보 파일을 특정 폴더에 업로드
4.2 ETL job 실행 eSWiTDB에 detail · summary 테이블 적재
4.3 최신 견적서 검색 (수동) 그룹웨어 Sales Quotation / SharePortal Archiving 직접 확인
4.4 가격정보 수동 입력 후 상신 eSWiT보상금액 = 가격 × warranty rate
4.5 Debit Note 인보이스 출력 · 송부 LDS Report site

변경 (TO-BE)

4.1 RMA_Details 파일 업로드 담당자파일서버기존과 동일
4.2 SNT 파일 다운로드 후 업로드 담당자본사 SNT 시스템Serial No, Item, Invoice No, Customer No, Warranty, Ship Date, Type
4.3 매크로 실행 담당자엑셀파일수동 검색 대체 — T8 필터 · Serial No 룩업 → Invoice No를 RMA_Details 미사용 컬럼에 업데이트 · SNT 타임스탬프 후 아카이빙 폴더 이동
4.4 ETL job 실행 담당자eSWiT기존과 동일
4.5 백엔드 Sales Price 검색쿼리 ITeSWiTInvoice No / Item 기준 자동 조회
4.6 신청화면 Sales Price 컬럼 추가 ITeSWiT보상금액 로직은 동일
4.7 Debit Note 인보이스 ITLDS Report siteInvoice No 컬럼 추가
담당자 체감 변화 — 견적서를 직접 찾아 입력하던 작업(4.3·4.4)이 사라지고, SNT 다운로드 + 매크로 실행만 하면 됩니다. 가격 조회·표시(4.5·4.6)와 양식 변경(4.7)은 IT에서 구축합니다.
보상금액 산식 — 보상금액 = 기준 단가 × warranty 경과기간별 rate
1~12개월 · 0.612~24개월 · 0.424~36개월 · 0.2DOA · 1.0
※ rate 로직은 변경 없이 그대로이며, 곱해지는 기준 단가만 견적가에서 판매가로 바뀝니다.

5

변경의 필요성과 기대효과

현행 방식의 문제점 네 가지는 이번 변경으로 각각 해소됩니다. 문제와 개선 효과를 짝지어 정리했습니다.

!
과다 보상 (핵심) — 환율과 HDD 시세 상승으로 최신 견적가가 판매 당시 가격을 웃돌아, 견적가 기준으로는 실제 판매가보다 많은 금액이 보상됨.
→ 판매 당시 인보이스 가격을 기준으로 전환하여 적정 보상 확보
!
수작업 의존 — 보상 아이템마다 담당자가 최신 견적서를 직접 검색해야 하고, 처리 품질이 담당자의 숙련도에 좌우됨.
→ 매크로 + 백엔드 자동 조회로 단순화되어 공수 절감
!
휴먼 에러 가능성 — 가격을 수동 입력하므로 오입력·누락·잘못된 견적 선택의 위험이 상존함.
→ 시스템이 Invoice No / Item 기준으로 자동 취득해 입력 오류 원천 차단
!
근거의 일관성·추적성 부족 — 견적서 출처가 분산되어 있고, 보상 단가가 판매 시점과 무관한 최신가라 정산·감사 근거가 약함.
→ 실제 판매 인보이스라는 단일 근거로 정산·감사 추적성 향상

예상 소요 시간

운영 공수는 건당, 구축 공수는 1회성으로 구분. 수치는 가정치로 회의에서 보정 필요.

약 5~10분/건		SNT 다운로드 (3분 이내) + 매크로 실행 (1분)
에러 정정수시 발생거의 없음
IT 구축 공수 (1회성, 가정)예상

백엔드 Sales Price 검색쿼리 (4.5)약 2~3일 (룰 확정 후)
신청화면 Sales Price 컬럼 추가 (4.6)약 1~2일
Debit Note 양식 수정 (4.7 — Invoice No 컬럼)약 0.5~1일
통합 테스트 · 데이터 검증약 2~3일
